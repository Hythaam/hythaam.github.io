Just a few years ago, "Prompt Engineering" was the fastest growing career in tech. (Probably; I have no data for this at all) Fast forward to today, and it's all but disappeared. So what happened? Models got smarter, agentic workflows were introduced, and recently, AI fluency has become a defactor requirement in the professional world instead of a niche role.

More specifically, having smarter models means that a lot of the weird tricks of yesteryear ("You are a genious software engineer. Have fun while completing this task") are no longer applicable. At the same time, the more serious research has moved on to solving task context hydration in the general case (LongRoPE context within LLMs, and RAG hydration in agentic systems).

The last point, that AI fluency has become the expectation rather than the exception, is worth expanding on. Models and agents may be better, but that doesn't mean prompting strategy is irrelevant. As AI use is becoming more central to professional work, it's more important than ever to know how the get the most out of an agent.

{/* truncate */}

This post is practical guidance for engineers and other AI agent users who already understand software systems but haven't yet built a strong mental model for how agentic LLM workflows actually function under the hood. The goal isn't to make you memorize vocabulary. It's to help you reason about the tradeoffs in your own workflow so you can get better results more consistently.

If there's anything to take away from this post, it's that **the model is only one part of the system.** The agent harness (the software that assembles context, advertises tools, and manages the interaction loop) shapes what the model sees, what it can do, and how it executes long-running tasks. Within the harness, context is the working memory that makes LLM inference both useful and expensive.

Before talking about cost or quality, though, lets briefly define the harness around the model.

---

## Harnesses: the runtime around the model

When people talk about using an AI agent, they usually focus on the model — Claude, GPT, Gemini, or whichever frontier system is in play. The application that actually enables users to do useful work with the model is the **harness**: the control layer that assembles context, exposes functional capabilities, and defines the mode of interaction between the user and model.

All of the concepts we'll discuss later (tools, skills, pruning, compaction) lives inside this layer. 

### The standard loop

The interaction between you, the harness, and the model follows an incredibly simple cycle:

1. Your request enters the harness.
2. The harness assembles the context window with the shared system instructions, conversation history, available tool definitions, and any retrieved information.
3. The harness sends this assembled context to the model via a provider API call.
4. The model processes the context and either answers directly, spends time thinking through a reasoning chain, or requests a tool call.
5. If a tool call is requested, the harness executes the tool and feeds the result back into context for another turn of the loop.
6. The loop ends when the model produces a final answer or the harness stops it (due to turn limits, token budgets, or other constraints).

:::note
How does the model signal that it's done? It doesn't "decide" to stop in the way a person would. Instead, the model is trained to generate a special stop token (or, in the case of tool-calling APIs, returns a response without any tool-use requests). The harness interprets this signal as the end of the loop and returns the response to the user.
:::

### Tools

Tools are advertised extensions that the model can invoke through the harness. Models are simple text prediction systems; things like searching a codebase, reading and writing files, executing code, querying a database, or making calls to an [MCP server](https://modelcontextprotocol.io/docs/getting-started/intro) require the model to produce a formatted _tool call block_ to request the harness to perform an action on it's behalf.

For the purpose of this discussion, the most important piece is that **tool outputs become part of the working context.** Every search result, file read, or API response the model requests gets appended to the conversation, consuming tokens and bringing more information to compete for the model's attention with everything else in the context window.

### Context management

As the loop runs, the conversation grows and tool results accumulate. The context continues to fill with each call and response. Harnesses manage this growth through two primary mechanisms:

- **Pruning** removes stale or low-value context. The harness will discard old tool outputs, trim earlier turns of conversation, or drop information that is no longer relevant to the current task.
- **Compaction** replaces a long history with a shorter retained state. Rather than keeping every turn of a long-running conversation, the harness can [summarize the conversation so far](https://platform.claude.com/docs/en/build-with-claude/compaction) into a condensed block that preserves the most important information while discarding the raw back-and-forth.

We'll return to these mechanisms in more detail later, but the important takeaway is that the harness has a few options for managing context when it grows beyond the model limits.

### Why this matters

The harness is the layer that controls what the model sees and what it can do; how the harness manages the context impacts the cost and quality of the result almost as much as the underlying model being used. Context is the entire working set of information that is processed on every turn of the loop, and its size, shape, and freshness determine both how much you pay for each request and the quality of the returned response.

## Costs: why context discipline matters even when subscriptions hide it

Model selection is the most apparent cost when using an AI model. Small or old models are cheaper, while large frontier models are expensive. While rates change depending on the model, the other component of billing is your token usage. Even though most providers makes token usage feel invisible, understanding what drives cost helps you reason about efficiency, which directly informs how far your workflow can stretch.

### Tokens: the billing unit

Every interaction with an LLM is measured in **tokens**. Tokens are the atomic units the model reads and writes.

:::note
A token is not quite a word and not quite a character. Depending on the model, one token averages roughly three-quarters of a word in prose. Code, logs, JSON, and structured outputs are often less token-efficient: a line of code that looks short may consume significantly more tokens than an equivalent-length sentence of natural language.
:::

This matters because long-running agent loops don't just send a prompt once. Each turn of the harness loop resends the accumulated context — system prompt, conversation history, tool definitions, and all prior tool outputs — plus whatever new information was added on the last turn. The model processes the full context on every turn, and you pay for it each time.

### Token pricing

Most provider APIs charge less for **input tokens** (what you send to the model) and more for **output tokens** (what the model generates back). Pricing also varies by capability tier. A frontier reasoning model costs more per token (both input and output) than a smaller, faster model.

The practical consequence is that a "cheap" model can still become expensive if you feed it bloated, undisciplined context on every turn that leads to low-quality, verbose output. Conversely, a more capable and expensive model used with lean, well-shaped context can end up costing less per task than a cheaper model drowning in noise.

:::note
It's not always huge difference, but how a model converts text into tokens can impact costs in ways that aren't immediately obvious. Anthropic's Opus 4.7 has the same per-token cost as Opus 4.6, but it actually uses ~35% more tokens for the same input.
:::

### Prompt caching

Provider APIs offer **prompt caching** to reduce cost and latency when the leading portion of the prompt stays stable across requests. When you send a request, their system checks if a prefix of your prompt matches something already cached from a recent call. If it does, the cached portion is reused at a [significantly reduced cost — often 90% cheaper than processing it fresh](https://platform.claude.com/docs/en/build-with-claude/prompt-caching).

Cache reuse rewards **stable prefixes**. Harnesses take advantage of this by keeping your system prompt, tool definitions, and other often-repeated context at the front of the prompt where they can be matched across requests. It can also reduce the cost of long-running conversations, where early parts of the context remain unchanged every round.

In practice, this means prompt construction isn't just about what information the model needs. It's also about structuring that information so the expensive parts stay cached.

:::note
#### Subsidized Inference

If you're using an AI agent through a higher-tier subscription plan, inefficient token usage might not feel like an urgent issue. The current flat-rate pricing and massive token limits encourage experimentation getting agents to perform low-value tasks without optimization.

Even so, cost-awareness is still a useful skill to build now. The current economics of inference are being shaped by competition and market-building. Long-term inference pricing will likely look different from the present moment. Understanding what drives cost means you'll be prepared to optimize when and where it matters, and you'll get better results in the meantime because the same decisions that reduce cost also tend to improve output quality.
:::

That brings us to the second half of the story. The same context decisions that drive token cost also affect whether the model can reliably use the information you give it — which can further drive up costs if more prompting is required to produce a satisfactory output.

## Output quality: how context impacts performance

It's tempting to assume that more context means better answers. After all, a model can't reason about information it doesn't have! But research and practice both show that this assumption has some important pitfalls. Larger available context windows do not automatically solve retrieval, salience, or reasoning problems even when all of the needed information is technically present.

### Fact retrieval degrades with context length

Models perform best when relevant facts are easy to locate in the prompt. In a short, focused prompt, the signal-to-noise ratio is high and the model can quickly find and use the information it needs.

As context grows, more competing noise is introduced. Long contexts create competition among instructions, examples, tool outputs, and retrieved documents. The model's attention (the mechanism it uses to determine which tokens are relevant to which) gets stretched across more and more material. [Research indicates](https://www.trychroma.com/research/context-rot) that even on intentionally simple retrieval tasks, models are less able to identify relevant answers within the context as length increases, often in non-uniform and surprising ways. Information can be present but not effectively usable.

### The missing middle

One of the most well-documented patterns in long-context behavior is the **"lost in the middle"** effect. [It was found](https://aclanthology.org/2024.tacl-1.9/) that language models tend to pay the most attention to information near the **beginning** and **end** of their input context, with significantly degraded performance for information placed in the middle, even for models explicitly designed for long contexts.

:::note
The "lost in the middle" effect is an area of active research. Newer models show some improvement, but the fundamental pattern remains relevant in practice, especially for the long agent traces and verbose tool outputs common in agentic workflows.
:::

Long agent workflows naturally accumulate material in the middle of the context. Early system instructions sit at the top. The most recent user message sits at the bottom. All of the prior turns, tool results, and intermediate reasoning pile up in exactly the region where the model is least likely to attend to it carefully.

### Context rot

Beyond positional effects, there is a broader phenomenon called **context rot**: the gradual decline in context quality over a long interaction. Context rot isn't just about raw length, it's the accumulation of low-value information (old assumptions, verbose tool outputs, contradictory instructions left over from previous turns).

[Anthropic has observed](https://www.anthropic.com/engineering/effective-context-engineering-for-ai-agents) that context must be treated as a finite resource with diminishing marginal returns. Every new token introduced depletes the model's "attention budget" by some amount.

### The upshot

These are all context-construction problems, and there are solutions to all of them available to users. So if long context can weaken retrieval and reasoning, the question becomes: how do you construct context so the right information stays salient?

## Context building: keeping the working set useful

Context building is how you respond to the problems described above. The goal is not maximum context, it's **useful context**: a working set small enough to keep the model focused and rich enough provide the information necessary for the task at hand.

Think of context building as an engineering activity. It's the process of designing the information environment the model will operate in, and the quality of that environment directly determines the quality of the output.

### System prompt

The system prompt is a durable instruction layer provided by whatever agent is being run by the harness. It anchors the model's behavior, identity, and constraints across every turn of the conversation. The system prompt remains stable at the front of the context window, where it benefits from both positional salience and prompt caching.

A well-crafted system prompt preserves key rules even as the rest of the context drifts. It's where you define tone, safety constraints, output format expectations, and the behavioral boundaries that define the role of the agent as a tool for the user.

:::note
The system prompt for most agent harnesses is hardcoded by the developer. However, some tools and platforms allow you to modify it to some degree — for example, through files like `CLAUDE.md` in Claude Code, custom instructions in ChatGPT, or system message fields in API-based applications. If your harness supports it, this is one of the highest-leverage points for improving output quality.

Most of the larger commercial applications will only allow you to append new sections to the system prompt, but some smaller open source projects allow it to be overwritten entirely.
:::

### Tools

Tools expand what the model can *do* — search, read, write, compute, query. Less obviously, tools pull targeted information *into* context. Tool calls allow the model to pull more data into the context on-the-fly, which enables agents to perform long-running tasks independantly.

Tool design matters for output quality. A well-designed tool returns scoped, well-shaped output that's easy for the model to use — a focused search result, a specific file section, a clean API response. A poorly designed tool dumps verbose, unstructured output into context, recreating exactly the retrieval and attention problems described in the previous section.

Since the model needs to be made aware of the available tools, each one has to be included in the model context. Having too many tools loaded or having tools that aren't applicable to the task at hand will increase the context length and reduce output quality.

### Skills

Skills are modular instruction or workflow packages that can be loaded only when necessary. Instead of stuffing every possible instruction into the system prompt, skills let the agent load specialized guidance only when it determines the guidance is relevant.

[Anthropic describes skills](https://www.anthropic.com/engineering/equipping-agents-for-the-real-world-with-agent-skills) as working through **progressive disclosure**: the harness will only load the name and description of each available skill into the context at first. When the model encounters a task that matches a skill, it instructs the harness to load the full instructions into context. If a skill contains even more detailed reference material, the agent can discover and load those additional files only as needed.

This design directly addresses the context efficiency issues that normal tools introduce. Instead of loading everything into the context at once, specific instructions and guidance stay out of the context until it's actually relevant, reducing background clutter and preserving the model's attention budget for the information that matters right now. In the same way that tools allow the model to load new working data into the context on the fly, skills essentially allow the model to load new tools.

:::note
Tools and skills are not commands. No amount of "YOU MUST DO THIS" will make an LLM as deterministic as a traditional program. Tools will go unused, skills will be misapplied, and workflow checklists will be ignored. Don't depend on an agent alone for any task that requires absolute adherence to policy, and _never_ provide it carte-blanche access to sensitive systems.
:::

### Agents and prompts

This is the user-configurable request framing layer where you describe what you want, how you want it, and what constraints apply. In many harnesses, this takes the form of custom agent configurations, persona definitions, or specialized prompt templates.

Better request framing reduces ambiguity and keeps the model focused. Vague instructions force the model to guess at intent or lead it down irrelevant paths that waste tokens _and_ pollute context with noise.

This is also where agent specialization becomes valuable. Rather than using one general-purpose agent with access to every tool and every domain, consider defining focused agents for different kinds of work:

- A **code review agent** that only has access to file reading and diff tools, keeping its context focused on the code under review.
- A **research agent** that has access to web search and document retrieval but doesn't need file-editing capabilities (and the associated context bloat)
- A **writing agent** that works with style guides and reference documents but doesn't need a code execution environment.

Each specialized agent carries a leaner context with fewer tool definitions, more focused instructions, and less noise.

### Subagents

Subagents are a way to allow an agent to isolate side work in an independent context instead of letting it flood the main conversation thread. When a task requires an investigation, a calculation, or a subsidiary piece of work, the harness can spin up a separate agent with its own context window to handle it. When the task is complete, the results come back without details of the intermediary steps.

Without subagents, every subsidiary task expands the main context — adding tool outputs, intermediate reasoning, and exploratory dead ends to a window that's already trying to track the primary task. With subagents, the main thread stays clean and focused.

Because each subagent starts with a fresh context window, multiple sub-agents can share the same cached context. And because subagents handle isolated workstreams, it's easy to switch to smaller, cheaper models for simpler tasks while reserving more capable models for complex tasks.

## Practical approaches: habits that improve outcomes without overengineering

The mechanisms above are how the harness manages context, but many of the most impactful improvements come from how *you* interact with the agent on a day-to-day basis (after all, agents are meant to imitate what a person can do).

### Be specific

When you give the model a task, include the goal, the constraints, any prior attempts, the expected output format, and anything you already know about the problem. Don't make the model guess at information you have.

This isn't just about being a "good prompter." It's about the retrieval problem. In a long context window, the model has to identify which information is relevant to the current task. Specific, well-structured instructions make the important facts easier to find and harder to bury under noise. Even if the model can technically access the information through it's own tools, providing it upfront (or just telling it to retrieve that data specifically) can dramatically reduce the number of tokens the model will use just to figure out that the data is necessary.

:::note
Remember that input tokens are cheaper than output tokens, so letting a model think for several paragraphs before it requests a tool call on it's own will be more expensive than just telling it to read the file
:::

Tell the model to clarify ambiguities rather than letting it silently guess. A clarifying question costs a few tokens, but will help prevent the model exploring an incorrect path.

### Treat the interaction like mentorship, not magic

A useful mental model: you're giving a scoped task to a capable but not omniscient collaborator. This collaborator is very fast, has broad _general_ knowledge, and can use tools, but it doesn't know your project history, your preferences, or your unstated constraints unless you tell it.

This means:

- **Give concrete, checkable tasks** rather than vague delegation. "Refactor the authentication module to use JWT tokens, keeping the existing session-based auth as a fallback" is better than "improve the auth system."
- **Break large tasks into smaller steps** rather than dumping an entire project specification into one prompt. Smaller tasks mean smaller contexts, which means better reasoning.
- **Review and redirect** rather than hoping the model gets everything right on the first try. Iteration is normal and expected.

### Compact or reset when context gets stale

When a conversation becomes long, repetitive, or stale (or when you've extracted the specific answers you needed from an earlier exchange) consider compacting or starting a new session. Most agent harnesses support this automatically or through an explicit action.

The goal of compaction is to preserve only what still matters: goals, decisions made, constraints, open questions, and source-of-truth facts. Everything else (the exploratory back-and-forth, verbose tool outputs, intermediate reasoning steps, etc.) gets summarized or discarded.

If the context has become stale, (whether because the task has drifted or the underlying data has changed since the last interaction) it may be better to start fresh than to rely on the model to compact the context. Use what you learned over the course of that interaction to provide a more concise and up-to-date context to the new session.

### Don't rely on the model to assemble all context for you

The harness does its best to build a useful context window, but you often know important facts that the harness does not. You know which files are relevant. You know what you tried yesterday. You know the business constraint the model has never been told about.

Participate actively in context selection and shaping. Paste in relevant code snippets. Link to the right documentation. State your constraints explicitly. The model is good at working *with* information — but it can only work with what it has.

If you suspect that there is some insight that's important in a pile of raw data, start a new session with another agent to extract the insight before handing it back to the main agent. It's better to start a new session than to let all of the raw data pollute the context for the rest of the task.

### Be deliberate about skill quality

If your harness supports skills or custom instructions, treat them as engineering artifacts. A good skill adds reusable procedure, structure, or domain judgment that makes the agent measurably more effective at a class of tasks.

Be skeptical of anything that adds tokens without adding much capability. A skill that injects 2,000 tokens of generic advice on every prompt is a net negative; the harm from increasing context length will almost certainly outweigh any marginal benefit the advice could provide. The best skills are focused, modular, and loaded only when they're relevant.

Conversely, a small skill that is only useful in highly-specific circumstances can also be harmful. The skill will be included in many contexts where it doesn't provide value, and when the skill _is_ relevant, the model may be able to accomplish the task without the skill anyway (albiet with an increased token count).

---

## Conclusion: context as a first-class engineering constraint

The model matters. But much of the behavior users observe — the quality of the output, the relevance of the reasoning, the efficiency of the interaction — comes from the harness and the context it assembles. A frontier model with poorly managed context will consistently underperform against a smaller model with well-curated context.

Better context management improves both cost and output quality. You send fewer tokens, which costs less. The tokens you do send are higher-signal, which means the model reasons more effectively.

Being specific in your prompts, breaking large tasks into smaller ones, compacting stale conversations, choosing the right tools, shaping what the model sees — these are practical skills that are rapidly becoming requirements across the professional world. As agents become more capable and the quality ceiling shifts, what you can do matters a lot less than what you can get a model to do.
